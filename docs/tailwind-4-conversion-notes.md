## Translation decisions

### Color naming — dual-layer preservation

The preset defines two distinct layers of color tokens:

1. **Brand-primitive layer** — keys like `bbl-primary`, `bbl-accent`, `bbl-bg`, `bbl-border`. These pointed at `--bbl-*` CSS custom properties in the companion CSS file. Utilities: `bg-bbl-primary`, `text-bbl-accent`, etc.

2. **Semantic/functional alias layer** — keys like `surface`, `surface-warm`, `text`, `heading`, `link`, `error-text`, `cta-bg`, etc. These pointed at `--color-*` CSS custom properties (which themselves alias the `--bbl-*` primitives).

In Tailwind 4's `@theme`, all color tokens share the `--color-*` namespace. Both layers are preserved:

- Brand primitives: `--color-bbl-primary`, `--color-bbl-accent`, etc.
- Functional aliases: `--color-surface`, `--color-text`, `--color-heading`, etc.

The utility class names generated by Tailwind 4 therefore match the preset's intent exactly: `bg-bbl-primary` (brand layer) and `bg-surface` (functional layer) are both available.

### Underlying CSS custom properties

Tailwind 4's `@theme` tokens are resolved at computed-value time by the browser, so color tokens declared as `var(--bbl-primary)` inside `@theme` correctly pick up whichever value `:root` or the dark-mode override has set. To support this, the file also declares a `:root` block with the raw `--bbl-*` and `--color-*` custom properties. These sit **outside** the `@theme` block and are not Tailwind tokens — they are the CSS foundation the tokens reference.

### Font size — line-height and letter-spacing

In Tailwind 3, `fontSize` entries accepted a two-element tuple `[size, { lineHeight, letterSpacing }]`. Tailwind 4's `--text-*` token holds only the font-size value; line-height and letter-spacing are separate namespaces (`--leading-*`, `--tracking-*`).

Decision: the per-step line-heights are emitted as `--leading-display`, `--leading-h1`, etc., and the per-step letter-spacings as `--tracking-display`, `--tracking-h1`, etc. This makes all three dimensions available as individual Tailwind utilities (`text-display`, `leading-display`, `tracking-display`) **and** as named CSS custom properties for base element styles in `global.css`.

No values were altered. The `h4` entry has no `letterSpacing` in the preset; none is added here.

### Spacing key names

The preset uses string keys `"sp-0"` through `"sp-10"` (with a hyphen). In Tailwind 4, spacing tokens are `--spacing-*`. The hyphen in the key is preserved: `--spacing-sp-0` through `--spacing-sp-10`. Utilities: `p-sp-4`, `mt-sp-7`, etc.

### Border radius key names

Preset keys are `"bbl-0"` through `"bbl-3"`. Tailwind 4 uses `--radius-*`. Result: `--radius-bbl-0` through `--radius-bbl-3`. Utilities: `rounded-bbl-1`, `rounded-bbl-2`, etc.

### Max-width → `--max-width-*` (corrected post-review)

In Tailwind 4, `max-w-*` utilities resolve from the `--max-width-*` namespace (primary), with `--spacing-*` and `--container-*` as fallbacks — confirmed by Tailwind's `utilities.ts` source. The two preset values (`measure: 70ch`, `container: 1200px`) are emitted as `--max-width-measure` and `--max-width-site`. The second token was renamed from `container` to `site` to avoid collision with Tailwind's built-in `container` utility class.

**History:** Agent 1 originally placed both tokens under `--width-*`, which generates `w-*` utilities (not `max-w-*`). The error was caught by Lars, verified by Agent 3 against the Tailwind 4 source code, and corrected in the main session before commit. Full source trail in the "Post-review corrections" section at the end of this file.

## Dark mode mechanism

The preset's colors are indirection tokens — they point at CSS custom properties (`--bbl-*`, `--color-*`). The light/dark swap happens by overriding those underlying custom properties under a dark-mode selector; the `@theme` tokens themselves never change. This mechanism is fully preserved.

The dark-mode selector in `tokens.css` is `.dark, [data-theme="dark"]`, matching the source `colors_and_type.css` exactly. TECH_STACK.md §4.2 describes `[data-theme="dark"]` as the preferred driver; the `.dark` class form is retained for parity with the design source and in case any component library uses it. There is no `@variant dark` in `tokens.css` — that belongs in `global.css` per TECH_STACK.md §4.2.

One dark-mode value is semantic, not a primitive flip:
- `--color-cta-text: #1C1C1E` — dark text on the gold CTA chip to avoid halation. Retained verbatim.

## Values flagged for reviewer attention

1. **`--color-text-muted` uses `color-mix()`** — the value `color-mix(in oklab, var(--bbl-neutral) 70%, transparent)` is a live CSS computation, not a hex literal. It is carried through unchanged. Confirm the target browser baseline supports `color-mix()` in oklab (broadly available since 2023).

2. **`--color-error-bg` uses `color-mix()`** — same situation: `color-mix(in oklab, var(--bbl-error) 10%, var(--bbl-bg))`. Carried through unchanged.

3. ~~**`--width-*` namespace for max-width**~~ — **RESOLVED.** Independently verified by Agent 3 (web search against Tailwind 4 source and docs); corrected to `--max-width-*` before commit. See "Post-review corrections" below.

4. **Self-referential `@theme` color tokens** — tokens like `--color-surface: var(--color-surface)` in the `@theme` block reference a same-named property in `:root`. This is intentional: `@theme` registers the token name in Tailwind's token table, while the actual value is resolved from the `:root` custom property by the browser at runtime. If this pattern causes issues with the specific Tailwind 4 build version, the alternative is to inline the raw hex values in `@theme` and duplicate the dark-mode block inside `@theme` using `@variant dark`. The current approach avoids duplication and preserves the single-source-of-truth for hex values.

5. **No `--font-face` declarations** — per task scope, `@font-face` blocks are excluded from `tokens.css` and will be added in `src/styles/global.css` separately.

6. **No base element styles** — `body`, `h1`–`h4`, `p`, `a`, `code`, etc. are excluded from `tokens.css` and will be handled in `global.css`.

## Post-review corrections

### 2026-04-23 — `maxWidth` namespace correction

**Issue.** Agent 1 placed the `measure` (`70ch`) and `container` (`1200px`) tokens under the `--width-*` namespace. Agent 2's verification accepted this as a "runtime-dependent" item requiring a smoke-test.

**Catch.** Lars flagged this before commit based on his own web search and did not accept the conversion as-is. He requested independent verification by a third sub-agent (Agent 3, Claude Sonnet) against primary Tailwind 4 sources before any file edits.

**Verification (Agent 3).** Agent 3 consulted Tailwind 4's source code at `packages/tailwindcss/src/utilities.ts` (tailwindlabs/tailwindcss), the official docs at `tailwindcss.com/docs/theme` and `tailwindcss.com/docs/max-width`, a maintainer statement in GitHub Discussion #16466 (@philipp-spiess, 2025-02-12: *"fine-grained controls exist: `--padding-*` for padding and `--max-width-*` for max-width utilities"*), and GitHub Discussions #17777 and #19020. Findings:

- `max-w-*` utilities resolve from the namespace list `['--max-width', '--spacing', '--container']` — in that priority order.
- `w-*` utilities resolve from `['--width', '--spacing', '--container']`.
- `--width-*` tokens therefore generate `w-*` utilities, **not** `max-w-*`.
- The built-in `--container-*` namespace also generates `max-w-*` utilities (as a lower-priority fallback), but is semantically loaded — it is simultaneously the namespace for container query variants (`@sm:*`) and the `container` utility class.

**Verdict (Agent 3):** Agent 1's `--width-*` choice is INCORRECT; the correct namespace is `--max-width-*`.

**Correction applied.** In `src/styles/tokens.css`:

| Before (incorrect) | After (corrected) | Generated utility |
|---|---|---|
| `--width-measure: 70ch;` | `--max-width-measure: 70ch;` | `max-w-measure` |
| `--width-container: 1200px;` | `--max-width-site: 1200px;` | `max-w-site` |

The second token was additionally renamed `container` → `site` on Agent 3's recommendation. "Container" collides with Tailwind's built-in `container` utility class and the `--container-*` namespace for container queries — semantically ambiguous. Agent 3's top-ranked alternative (`site`) was chosen; `page`, `main`, and `layout` were also viable. If downstream code prefers one of those, it's a trivial rename.

**Primary citation.** `https://raw.githubusercontent.com/tailwindlabs/tailwindcss/main/packages/tailwindcss/src/utilities.ts` — the exact array literal establishing the namespace resolution order for each utility prefix.

**Process note.** The three-agent pipeline (Agent 1 convert → Agent 2 verify → Agent 3 namespace challenge) is what caught this. Agent 2 flagged the token as "version-dependent" without independently verifying the documentation; Agent 3's targeted web search against primary sources closed the gap. Future translation tasks that involve Tailwind 4 namespace decisions should include the Agent 3 style of primary-source verification as part of the verification loop, not as a post-hoc escalation.
